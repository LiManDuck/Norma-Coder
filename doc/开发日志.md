## 修复 MCP 服务器崩溃时挂起请求等满 60s 超时
- 审计 `mcp/client.py` 的 `_read_loop` 发现健壮性缺陷：当 MCP 服务器子进程崩溃或关闭 stdout 时，`_read_loop` 在 EOF 处 `break` 退出，但 `_pending_requests` 中的 future **永远不会被 resolve**--调用方需等满 `_send_request` 的 60s 超时才感知连接已断。对 codeagent CLI，MCP 服务器崩溃（配置错误/服务器 bug）是真实场景，60s 挂起体验很差。
- 修复：`_read_loop` 增加 `finally` 块，在连接断开（EOF/崩溃/取消）时调用新增的 `_fail_pending(exc)`，把所有挂起 future 立即置为 `ConnectionError`，使在途请求快速失败而非等满超时。
- 回归 `test_mcp_stdio.py` 新增 `_run_crash`（mock 服务器 `crash` 工具 `sys.exit(1)` 不响应）：调用 crash 工具后服务器退出 -> read_loop EOF -> `_fail_pending` -> `call_tool` 抛 ConnectionError -> `MCPTool.execute` 捕获为 is_error=True，断言 `elapsed < 30s`（无此修复会等满 60s 才失败，该断言正是验证修复生效）。
- 全量回归 18/18 绿。
